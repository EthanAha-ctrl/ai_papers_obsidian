# RPCS3 arm64 架构支持：技术深度解析

这篇文章宣布了 **RPCS3**（PlayStation 3 模拟器）正式支持 **arm64** 架构，可在 Linux、macOS 和 Windows arm64 设备上原生运行。下面我从**第一性原理**出发，详细剖析其技术内涵。

---

## 1. 背景：为什么 arm64 支持如此重要？

### 1.1 历史时间线

| 时间节点 | 里程碑 |
|---------|--------|
| 2021年 Apple M1 发布 | 启动 arm64 探索 |
| 2021年12月 | Nekotekina 完成 arm64 基础编译（仅解释器，无法运行游戏） |
| 2022年1月 | kd-11 修复 RSX 图形，首次跑通 samples |
| 2023年末 | 仍无 LLVM JIT 支持 |
| 2024年6月 | 商业游戏首次在 arm64 LLVM 上运行 |
| 2024年8月 | 正式发布 arm64 支持 |

### 1.2 核心挑战的本质

PS3 的 Cell 处理器架构极其特殊：
- **PPU** (PowerPC Processing Element)：主处理器，基于 PowerPC ISA
- **SPU** (Synergistic Processing Element)：8个协处理器，专用 SIMD 架构

**模拟的本质是 ISA 转换**：`PPU/SPU 指令 → 宿主机指令`

这需要**动态二进制翻译**（Dynamic Binary Translation, DBT），而 **LLVM JIT** 是性能的关键。

## 2. LLVM Recompiler 的 arm64 适配：核心难题

### 2.1 问题根源：x86 与 arm64 的调用约定差异

#### x86-64 调用模型

```
┌─────────────────────────────────────────────────────────┐
│  x86-64 Stack-based Call Chain                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   ┌──────────┐                                          │
│   │ caller   │                                          │
│   └────┬─────┘                                          │
│        │ call func_A     ───→  push return_addr on stack│
│   ┌────▼─────┐                     ┌──────────────┐    │
│   │ func_A   │                     │  Stack       │    │
│   └────┬─────┘                     │ ┌──────────┐ │    │
│        │ call func_B               │ │ ret_B    │ │    │
│   ┌────▼─────┐                     │ ├──────────┤ │    │
│   │ func_B   │                     │ │ ret_A    │ │    │
│   └────┬─────┘                     │ ├──────────┤ │    │
│        │ ret                       │ │ ret_main │ │    │
│   ┌────▼─────┐                     │ └──────────┘ │    │
│   │ unwind   │←── ret chain pops   └──────────────┘    │
│   └──────────┘                                          │
│                                                         │
│   关键：ret 指令从栈中弹出返回地址，可链式返回          │
└─────────────────────────────────────────────────────────┘
```

#### arm64 调用模型

```
┌─────────────────────────────────────────────────────────┐
│  arm64 Register-based Linking (LR = X30)                │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   ┌──────────┐                                          │
│   │ caller   │                                          │
│   └────┬─────┘                                          │
│        │ bl func_A    ───→  LR = return_addr (单寄存器) │
│   ┌────▼─────┐                                          │
│   │ func_A   │         ┌─────────────────┐             │
│   └────┬─────┘         │  LR Register    │             │
│        │ bl func_B     │  (X30)          │             │
│   ┌────▼─────┐         │  ┌───────────┐  │             │
│   │ func_B   │ ────────┼──│ ret_B (!)│  │             │
│   └────┬─────┘         │  │ (被覆盖!) │  │             │
│        │ ret           │  └───────────┘  │             │
│   ┌────▼─────┐         └─────────────────┘             │
│   │ ❌问题   │                                          │
│   │ ret 指令跳转到 LR (ret_B)                           │
│   │ 但 func_A 的返回地址已被覆盖！                       │
│   │ 结果：无限循环或崩溃                                 │
│   └──────────┘                                          │
└─────────────────────────────────────────────────────────┘
```

**数学化表达**：

设调用链为 $C = \{f_0 \rightarrow f_1 \rightarrow f_2 \rightarrow ... \rightarrow f_n\}$

- **x86-64**：返回地址存储在栈中，形成序列 $R_{stack} = [r_n, r_{n-1}, ..., r_0]$，通过 `ret` 链式弹出
- **arm64**：返回地址存储在单一寄存器 $LR$，每次 `bl` 指令会**覆盖** $LR$

$$LR_{new} = PC + 4 \quad \text{(bl 指令自动设置)}$$

问题：**tail-call optimized frames**（尾调用优化帧）在 arm64 上无法使用传统的 ret-chain！

### 2.3 尝试的解决方案对比

| 方案 | 描述 | 优点 | 缺点 | 结果 |
|------|------|------|------|------|
| **Shadow Stack** | 为每次调用分配最小栈空间，模拟 x86 行为 | 兼容性好 | 基本块会覆盖返回地址，导致随机崩溃 | ❌ |
| **Register Save** | 保留寄存器作为 shadow call stack pointer | 性能较好 | 需要修改 LLVM，维护成本高 | ❌ |
| **Far Jump Gateway** | 所有 return 替换为远跳转到 host gateway | 干净分离 | JIT 块间循环和递归导致性能下降严重 | ❌ |
| **IR Transformer** ✓ | 分析并转换 IR，适配 arm64 要求 | 无需修改 LLVM，可维护 | 需要深入理解 IR 语义 | ✅ |

### 2.4 最终方案：IR Transformer 的技术细节

```
┌────────────────────────────────────────────────────────────────┐
│                    IR Transformer 工作流程                     │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│   Step 1: 生成 x86-64 IR (Reference Architecture)             │
│   ┌─────────────────────────────────────────────┐             │
│   │ define @jit_block_0x1234(%ctx) {            │             │
│   │   %1 = load i32 %ctx, offset=0x100         │             │
│   │   %2 = add %1, 1                           │             │
│   │   store %2, %ctx, offset=0x100             │             │
│   │   tail call @jit_block_0x5678(%ctx)  ←── tail call │     │
│   │   ret void                                  │             │
│   │ }                                           │             │
│   └─────────────────────────────────────────────┘             │
│                         │                                      │
│                         ▼                                      │
│   Step 2: IR Transformer 分析                                  │
│   ┌─────────────────────────────────────────────┐             │
│   │  分析点：                                    │             │
│   │  1. 识别 tail call 边界                     │             │
│   │  2. 标记 "leaf nodes" (无外部调用的块)      │             │
│   │  3. 追踪 host gateway 位置                  │             │
│   │  4. 确定 ABI 寄存器分配                     │             │
│   └─────────────────────────────────────────────┘             │
│                         │                                      │
│                         ▼                                      │
│   Step 3: 生成 arm64 IR                                       │
│   ┌─────────────────────────────────────────────┐             │
│   │ define @jit_block_0x1234_arm(%ctx) {        │             │
│   │   ; x0 = %ctx (guest thread context)        │             │
│   │   %1 = load i32 %ctx, offset=0x100         │             │
│   │   %2 = add %1, 1                           │             │
│   │   store %2, %ctx, offset=0x100             │             │
│   │                                             │             │
│   │   ; 转换 tail call 为 direct branch        │             │
│   │   ; 不使用 bl (避免覆盖 LR)                │             │
│   │   b @jit_block_0x5678_arm  ←── 直接跳转    │             │
│   │ }                                           │             │
│   └─────────────────────────────────────────────┘             │
│                                                                │
│   关键洞察：                                                   │
│   - 不需要保存完整调用链                                       │
│   - 只需知道最近的 host gateway 位置                          │
│   - ABI 约定：x0 = guest thread context                       │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

## 3. 16K 内存页问题：隐藏的性能杀手

### 3.1 问题本质

```
┌─────────────────────────────────────────────────────────────┐
│                    Memory Page Size 对比                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   x86 / PS3: 4KiB pages                                    │
│   arm64 (大多数): 16KiB pages                               │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐  │
│   │           4KiB Page (4096 bytes)                    │  │
│   │  ┌────┬────┬────┬────┬────┬────┬────┬────┐         │  │
│   │  │ T1 │ T2 │ T3 │ T4 │ T5 │ T6 │ T7 │ T8 │         │  │
│   │  │4KB │4KB │4KB │4KB │4KB │4KB │4KB │4KB │         │  │
│   │  └────┴────┴────┴────┴────┴────┴────┴────┘         │  │
│   │     ↑                                               │  │
│   │     单个小纹理只需 1 page                           │  │
│   └─────────────────────────────────────────────────────┘  │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐  │
│   │           16KiB Page (16384 bytes)                  │  │
│   │  ┌─────────────────┬─────────────────┐              │  │
│   │  │      T1         │       T2        │              │  │
│   │  │     16KB        │      16KB       │              │  │

│   │     ↑                    ↑                          │  │
│   │     即使 T1 只有 1KB，也占用 16KB page              │  │

│   └─────────────────────────────────────────────────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 Dirty Page Tracking 的性能影响

**公式化分析**：

设纹理 $T$ 的大小为 $S_T$，页面大小为 $P$，需要的页面数为：

$$N_{pages} = \lceil \frac{S_T}{P} \rceil$$

每次 dirty page eviction 的开销：

$$Cost_{evict} = N_{pages} \times (T_{read} + T_{upload})$$

其中：
- $T_{read}$：从内存读取一页的时间
- $T_{upload}$：上传到 GPU 的时间

**问题**：PS3 的纹理/GPU 对象通常很小：
- Texel buffers
- Uniform buffers  
- Shader storage buffers

当 $S_T \ll P_{16K}$ 时，浪费倍数为：

$$Waste\_Factor = \frac{P}{S_T} \approx \frac{16384}{S_T}$$

对于 512 字节的对象：$Waste\_Factor = 32x$！

```
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  渲染负载 ≈ 像素数量                                        │
│                                                             │
│  像素数量公式：                                              │
│  $$P = W \times H$$                                         │
│                                                             │
│  不同分辨率的像素数量对比：                                  │
│  ┌──────────────┬────────────┬────────────┬──────────────┐ │
│  │  分辨率       │  像素数量   │  相对负载  │  FPS 示例    │ │
│  ├──────────────┼────────────┼────────────┼──────────────┤ │
│  │  720p        │  921,600   │  100%      │  ~10 FPS     │ │
│  │  480p        │  307,200   │  33%       │  ~20 FPS     │ │
│  │  360p        │  230,400   │  25%       │  ~25 FPS     │ │
│  │  272p (PSP)  │  147,968   │  16%       │  ~30 FPS ✓   │ │
│  └──────────────┴────────────┴────────────┴──────────────┘ │
│                                                             │
│  降级公式：                                                  │
│  $$Scale_{factor} = \sqrt{\frac{P_{target}}{P_{original}}}$$│
│                                                             │
│  对于 PSP 分辨率：                                          │
│  $$Scale = \sqrt{\frac{147968}{921600}} = \sqrt{0.16} = 0.4$$│
│                                                             │
│  实际设置：38% (PR #15977)                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 7. 技术洞察总结

### 7.1 第一性原理视角

**模拟器的本质**：
$$Emulation = Translation(ISA_{guest}) \rightarrow Execution(ISA_{host})$$

关键性能因素：
1. **翻译效率**：LLVM IR → 机器码的质量
2. **内存模型**：页大小、对齐、缓存一致性
3. **调用约定**：ABI 匹配程度
4. **GPU 映射**：图形 API 的能力差异

## 总结

这篇文章详细记录了 **RPCS3 从 x86-centric 设计到 arm64 原生支持**的技术历程。核心创新是 **IR Transformer** 方案，它：

1. **避免了维护 LLVM fork 的负担**
2. **保持了与现有 x86-64 后端的兼容性**
3. **通过 IR 层面的分析和转换，优雅解决了 arm64 调用约定问题**

同时，文章也揭示了**跨架构模拟的深层挑战**——不仅是指令翻译，还有内存模型、GPU 驱动、操作系统差异等多维度问题。树莓派 5 的成功运行更是证明了**软件优化可以弥补硬件差距**，将 PS3 模拟带入了前所未有的便携设备领域。
