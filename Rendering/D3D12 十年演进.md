# D3D12 十年演进：API 与开发范式的全面回顾

这是一篇由 TheRealMJP 撰写的技术博客，系统性地回顾了 Direct3D 12 自 2015 年 Windows 10 发布以来十年间的所有重要更新。文章不仅罗列了 API 层面的变化，还分享了作者个人编程风格的演变。

---

## 一、核心 D3D12 API 新增功能

### 1. **Programmable Sample Points（可编程采样点）**

**核心概念**：允许开发者自定义 MSAA 的子采样点位置，而非使用硬件预设的旋转网格模式。

```
传统 MSAA:  4x 标准采样模式（旋转网格）
              ×   ×
                ×   ×
自定义采样: 开发者可指定任意位置
```

**技术细节**：
- API 可指定覆盖 2×2 像素四边形范围内的采样位置
- 主要用途：
  1. 生成更不规则的采样模式，改变抗锯齿特性
  2. **软件实现 Variable Rate Shading (VRS)**：生成半分辨率深度缓冲时，使深度子采样位置精确匹配全分辨率 pass

**公式表达**：
$$\text{SamplePosition}(x, y) = \left( \frac{x}{16}, \frac{y}{16} \right) \in [0, 1)^2$$

其中 x, y 为 16 级精度坐标，覆盖一个 2×2 像素块。

---

### 2. **View Instancing（视图实例化）**

**核心概念**：为 VR/立体渲染设计的单 pass 多视图渲染机制。

**架构图**：
```
┌─────────────────────────────────────────┐
│           Single Draw Call              │
├─────────────────────────────────────────┤
│  SV_ViewID = 0 → Viewport 0 / Slice 0   │  (左眼)
│  SV_ViewID = 1 → Viewport 1 / Slice 1   │  (右眼)
│  SV_ViewID = 2 → Viewport 2 / Slice 2   │  (可选)
│  SV_ViewID = 3 → Viewport 3 / Slice 3   │  (可选)
└─────────────────────────────────────────┘
```

**技术要点**：
- 最多支持 4 个视图
- Shader 中通过 `SV_ViewID` 辨识当前视图
- 可配置每个视图映射到不同 viewport 或纹理数组切片
- 支持每视图独立剔除（通过 bitfield mask）

**硬件支持差异**：
| 厂商 | Tier | 实现方式 |
|------|------|----------|
| NVIDIA | Tier 2+ | Viewport Broadcast 硬件加速 |
| AMD | Tier 1 | Draw-call 级别循环 |

**实际建议**：由于 AMD 仅支持 Tier 1，使用传统实例化 + `SV_RenderTargetArrayIndex` / `SV_ViewportArrayIndex` 可能更优。

---

### 3. **Depth Bounds Test（深度边界测试）**

**历史背景**：Doom 3 时代的模板阴影优化技术，终于在 D3D12 中正式暴露。

**工作原理**：
```
对于每个像素：
  读取现有深度值 Z_existing（非当前图元深度）
  检查: Z_min ≤ Z_existing ≤ Z_max ?
  如果在范围内 → 通过深度测试
  否则 → 剔除
```

**用途场景**：
1. 老式 Deferred Rendering 的光源包围体渲染
2. Deferred Decals（包围盒光栅化）
3. **PS3 时代的级联阴影技巧**：设置深度边界为当前 cascade 的 min/max，绘制全屏三角形

---

### 4. **Variable Rate Shading (VRS，可变速率着色)**

**核心思想**：通过降低像素着色器执行频率来提升性能，同时保持全分辨率深度测试。

**Shading Rate 定义**：
```
Shading Rate = 1x1  → 每像素执行一次 PS（标准）
Shading Rate = 2x1  → 每 2×1 像素执行一次 PS
Shading Rate = 2x2  → 每 2×2 像素执行一次 PS
...
Shading Rate = 4x4  → 每 4×4 像素执行一次 PS
```

**三种控制方式**：

| 方式 | 控制粒度 | 使用场景 |
|------|----------|----------|
| Per-Draw | 命令列表设置 | 手动标记半分辨率透明/粒子 |
| Per-Primitive | Shader 输出 `SV_ShadingRate` | 动态每三角形控制 |
| Per-Tile | 纹理图 (最大 16×16 tile) | 基于前一帧内容的自适应分析 |

**性能权衡公式**：
$$\text{ActualSpeedup} = \frac{N_{ps\_original}}{N_{ps\_reduced} + \text{WaveFillOverhead} + \text{BroadcastCost}}$$

其中：
- $N_{ps\_original}$：原始像素着色器线程数
- $N_{ps\_reduced}$：减少后的像素着色器线程数
- $\text{WaveFillOverhead}$：Wave 未满时的开销
- $\text{BroadcastCost}$：粗粒度像素广播到实际像素的开销

**注意事项**：
1. 硬件仍以 2×2 quad 启动 PS，需填满 wave 才能执行
2. 低分辨率区域采用点采样广播，需要额外空间滤波或时域重建

---

### 5. **Relaxed Format Casting（宽松格式转换）**

**D3D11/D3D12 早期规则**：
```
创建纹理：R16_TYPELESS
创建视图：R16_UINT SRV
创建视图：R16_UNORM SRV
```

**新规则**：
```
创建纹理：R16_UNORM (typed)
创建视图：R16_UINT SRV (兼容格式)
         ↑ 无需 TYPELESS！
```

**限制**：不能在浮点与非浮点格式之间转换，避免驱动禁用优化。

**对于深度缓冲**：现在可以直接创建 typed 深度格式，并通过 SRV 读取，无需 TYPELESS。

---

### 6. **Directly Writing To Block-Compressed Formats（直接写入 BC 格式）**

**历史问题**：GPU 只能通过 Copy 写入 BC 纹理。

**D3D10 "后门"**：
```
BC1 纹理 (4x4 block, 8 bytes)
    ↑ copy from
R32G32_UINT 纹理 (1/4 width, 1/4 height)
```

**D3D12 Enhanced Barriers 方案**：

创建纹理时声明可转换格式：
```cpp
D3D12_RESOURCE_DESC desc = {};
desc.Format = DXGI_FORMAT_BC1_UNORM;
desc.CastableFormats = { DXGI_FORMAT_R32G32_UINT };  // 8 bytes/block
```

| BC 格式 | 对应 UINT 格式 | 每块字节数 |
|---------|----------------|------------|
| BC1, BC4 | R32G32_UINT | 8 |
| BC2, BC3, BC5, BC6, BC7 | R32G32B32A32_UINT | 16 |

---

### 7. **Writable R9G9B9E5 Textures**

**格式解析**：
```
R9G9B9E5_SHAREDEXP (32 bits total):
┌─────────┬─────────┬─────────┬─────────┐
│ R (9b)  │ G (9b)  │ B (9b)  │ E (5b)  │
└─────────┴─────────┴─────────┴─────────┘
       Mantissa bits        Shared Exponent

解码公式：
  Value = Mantissa × 2^(Exponent - 15)
```

**与 R11G11B10_FLOAT 对比**：

| 格式 | R bits | G bits | B bits | Exp bits |
|------|--------|--------|--------|----------|
| R11G11B10_FLOAT | 6+5 | 6+5 | 5+5 | 每通道独立 |
| R9G9B9E5 | 9 | 9 | 9 | 5 (共享) |

**优势**：更多 mantissa bits，减少 banding。
**限制**：共享指数，适合三通道幅度高度相关的场景。

**硬件支持**：AMD RDNA2+。

---

### 8. **WriteBufferImmediate + OpenExistingHeapFromAddress**

**用途**：GPU 崩溃调试的"面包屑"机制。

**工作流程**：
```
命令列表录制：
  WriteBufferImmediate(buffer, value_1)  ← "面包屑 1"
  Dispatch(...)
  WriteBufferImmediate(buffer, value_2)  ← "面包屑 2"
  Draw(...)
  ...

Device Removed 后：
  OpenExistingHeapFromAddress(buffer_addr)
  检查 buffer 内容 → 确定最后到达的"面包屑"
```

---

### 9. **GPU Upload Heaps（GPU 上传堆）**

**背景问题**：
```
传统架构：
┌─────────────┐      PCIe      ┌─────────────┐
│   SysRAM    │ ─────────────→ │    VRAM     │
│  (CPU 可见) │   GPU Copy    │  (GPU 专用) │
└─────────────┘                └─────────────┘

Resizable BAR (ReBAR)：
┌─────────────┐                ┌─────────────┐
│   SysRAM    │                │    VRAM     │
│             │   全部映射到   │   ↑         │
│             │ ← CPU 地址空间 │   CPU 可见！│
└─────────────┘                └─────────────┘
```

**新 API**：
```cpp
D3D12_HEAP_TYPE_GPU_UPLOAD  // 新枚举值
```

**使用方式**：
- Buffer：直接 `memcpy`
- Texture：使用 `WriteToSubresource`（因为布局不透明）

**硬件支持**：
| 厂商 | 起始架构 |
|------|----------|
| AMD | RDNA2+ |
| NVIDIA | Ampere+ |
| Intel | 全部独显 |

**限制**：仅 Windows 11（需要内核改动）。

---

### 10. **ExecuteIndirect Incrementing Constant**

**问题**：Vulkan 有 `DrawID`，D3D12 缺失。

**旧方案**：在 Command Signature 中设置 root constant → 驱动实现复杂。

**新方案**：
```hlsl
// Shader 中
uint DrawID;  // 隐式递增的常量缓冲值

// 每个 Draw/Dispatch 自动递增
```

**支持情况**：NVIDIA Pascal+ 支持 Tier 1.1。

---

### 11. **Sampler Feedback（采样器反馈）**

**核心功能**：追踪纹理采样时实际访问的 texel 区域。

**用途**：
1. Sparse Virtual Texture 的驻留判断
2. Texture-Space Shading 的 texel 定位

**API 流程**：
```
1. 创建 Feedback 资源（不透明格式）
2. 绑定额外描述符
3. Shader 中添加反馈生成
4. Resolve 到可读格式
5. Copy 回 CPU（如需）
```

**缺点**：API 复杂，性能开销可能高于原子写入方案。

---

### 12. **Tight Resource Alignment（紧凑资源对齐）**

**问题**：
```
D3D12_DEFAULT_RESOURCE_PLACEMENT_ALIGNMENT = 64 KB
→ 创建大量小 buffer 时内存浪费严重
```

**新方案**：
```cpp
D3D12_RESOURCE_FLAG_USE_TIGHT_ALIGNMENT

Buffer: 8B ~ 256B 对齐
Texture (非 MSAA): 8B ~ 64KB 对齐
Texture (MSAA): 8B ~ 4MB 对齐
```

**状态**：预览阶段（Agility SDK 1.716.1）。

---

### 13. **Render Passes**

**目标硬件**：Tile-Based GPU（如 Qualcomm Adreno）。

**API 替换**：
```
旧：OMSetRenderTargets(...)
新：BeginRenderPass(...) / EndRenderPass()
```

**关键参数**：
```cpp
D3D12_RENDER_PASS_BEGINNING_ACCESS  // 首次访问行为
D3D12_RENDER_PASS_ENDING_ACCESS     // 结束访问行为
```

**优化原理**：
- 告知驱动渲染目标数据的生命周期
- 避免不必要的 on-chip memory 加载/存储
- 减少带宽消耗

---

### 14. **Enhanced Barriers（增强屏障）**

**D3D12 Legacy vs Vulkan 对比**：

```
D3D12 Legacy:
  ResourceBarrier(Transition: STATE_A → STATE_B)
  └→ 驱动推断需要的同步、缓存刷新、布局转换

Vulkan:
  vkCmdPipelineBarrier(
    srcStageMask, dstStageMask,    // 同步
    memoryBarrierCount,            // 缓存刷新
    imageMemoryBarrierCount        // 布局转换
  )
```

**Legacy 问题**：
1. "Promotion" 和 "Decay" 概念混乱
2. 隐式同步难以理解
3. 驱动可能插入过多刷新

**Enhanced Barriers 方案**：
```cpp
D3D12_BARRIER_SYNC       // 同步点
D3D12_BARRIER_ACCESS     // 访问类型
D3D12_BARRIER_LAYOUT     // 纹理布局
```

**额外好处**：
- 可在 barrier 中 discard 资源
- 避免驱动的额外刷新

---

### 15. **Ray Tracing (DXR)**

**两种使用模式**：

**DXR 1.0 - DispatchRays Pipeline**：
```
RayGen Shader
    │
    ├─→ TraceRay() ─→ Hit? ─→ ClosestHit Shader
    │                      └→ AnyHit Shader (可选)
    │
    └─→ Miss? ─→ Miss Shader
```

**DXR 1.1 - RayQuery (Inline Tracing)**：
```hlsl
RayQuery<RAY_FLAG_NONE> q;
q.TraceRayInline(...);
while(q.Proceed()) {
  // 完全由用户控制
}
```

**加速结构**：
```
Bottom Level AS (BLAS)
├─ Mesh A (三角形数据)
├─ Mesh B
└─ ...

Top Level AS (TLAS)
├─ Instance 0 → BLAS A (Transform M0)
├─ Instance 1 → BLAS A (Transform M1)  // 复用相同 mesh
└─ Instance 2 → BLAS B
```

**动态 Mesh 处理**：
| 方案 | 速度 | 质量 |
|------|------|------|
| Full Rebuild | 慢 | 最优 |
| Refit | 快 | 可能退化 |
| Refit + 周期性 Rebuild | 平衡 | 推荐 |

**DXR 1.2 - Shader Execution Reordering (SER)**：
- 按材质 ID 等键重新打包 Wave
- 减少分支发散

---

### 16. **Mesh Shaders**

**传统管线**：
```
IA → VS → HS → DS → GS → RS → PS
      └── 固定功能阶段 ──┘
```

**Mesh Shader 管线**：
```
Amplification Shader (可选)
        ↓ DispatchMesh()
Mesh Shader (Thread Group)
        ↓
   输出：Vertices + Indices
        ↓
     RS → PS
```

**Mesh Shader 特性**：
- Compute-like 编程模型
- 使用 `groupshared` 内存协作
- 无固定功能 vertex fetch
- 可程序化生成几何

**Meshlet 典型配置**：
```
每个 Meshlet:
  - 最多 64 vertices
  - 最多 126 triangles (NVIDIA) / 256 triangles (AMD)
```

**Amplification Shader 用途**：
```hlsl
// Culling + LOD 选择
if (MeshletVisible(meshletID)) {
  DispatchMesh(1, 1, 1, payload);
}
```

**注意事项**：
- AMD: Amplification Shader 剔除过多时会因 command processor ring buffer 填满而性能下降
- 不同 GPU 对 vertex/triangle 输出计数敏感度不同

---

### 17. **Work Graphs**

**核心概念**：GPU 自驱动的工作调度图。

**对比 ExecuteIndirect**：
```
ExecuteIndirect:
  CPU → 准备参数缓冲
  GPU → 读取参数，执行 Draw/Dispatch

Work Graphs:
  GPU Node A → Launch(Node B, grid)
           → Launch(Node C, grid)
           → ...
```

**示例场景**：
```
Node: TileClassification
  │
  ├─→ Launch(Node: DeferredShading, grid)
  ├─→ Launch(Node: ForwardShading, grid)
  └─→ Launch(Node: SpecialPass, grid)
```

**限制**：
- 图拓扑需静态确定
- 无跨节点递归（仅可递归到自身）

**Mesh Nodes**：预览功能，允许在 Work Graph 中直接输出三角形。

**性能考量**：
- 简单场景可能被 ExecuteIndirect 超越
- 复杂场景可能获得性能和内存优势

## 二、新 Shader 特性

### 1. **Wave Ops**

**基本概念**：
```
Wave = Warp (NVIDIA, 32 threads)
     = Wavefront (AMD, 32 or 64 threads)
     = Subgroup (Vulkan 术语)
```

**主要操作**：

| 操作类型 | 函数 | 说明 |
|----------|------|------|
| Broadcast | `WaveReadLaneFirst` | 读取 lane 0 的值 |
| | `WaveReadLaneAt(i)` | 读取指定 lane 的值 |
| Ballot | `WaveActiveBallot` | 返回条件 mask |
| | `WaveActiveAllTrue` | 全部为真？ |
| | `WaveActiveAnyTrue` | 任一为真？ |
| Reduction | `WaveActiveSum` | 全 Wave 求和 |
| Scan | `WavePrefixCountBits` | 前缀和 |

**前缀和公式**：
$$\text{WavePrefixSum}(i) = \sum_{j=0}^{i-1} \text{value}_j$$

**Scalarization 应用**：
```hlsl
// 向量化的 uniform 数据
float4 uniformData = WaveReadLaneFirst(laneData);
// 所有 lane 使用相同数据 → 标量寄存器
```

**Wave Size 差异**：

| 厂商 | Wave Size |
|------|-----------|
| NVIDIA | 32 (固定) |
| AMD (GCN/Vega) | 64 |
| AMD (RDNA+) | 32 或 64 |
| Intel | 8/16/32 |

---

### 2. **Quad Ops**

**概念**：操作 2×2 quad 内的 4 个像素着色器线程。

```hlsl
QuadReadLaneAt(value, quadLaneID)
QuadAcrossX(value)   // 水平相邻
QuadAcrossY(value)   // 垂直相邻
```

**应用**：Quad Overshading 可视化等。

---

### 3. **16-bit Floats and Integers**

**类型**：
```hlsl
float16_t, float16_t2, float16_t3, float16_t4
int16_t, uint16_t
```

**优势**：
- 更紧凑的数据打包
- 可能的 ALU 吞吐量提升（需 packed 指令）
- 减少寄存器压力

**AMD 特殊情况**：
需要使用 packed 指令同时对两组 fp16 输入执行操作才能获得更高吞吐。

**陷阱**：容易引入隐式 fp16 ↔ fp32 转换。

---

### 4. **Barycentrics（重心坐标）**

**定义**：
三角形内任意点的重心坐标 $(\lambda_0, \lambda_1, \lambda_2)$：
$$P = \lambda_0 V_0 + \lambda_1 V_1 + \lambda_2 V_2$$
$$\lambda_0 + \lambda_1 + \lambda_2 = 1$$

**用途**：
1. 自定义属性插值（可压缩打包）
2. **抗锯齿线框渲染**（无需额外 pass 或 Geometry Shader）

---

```
2. 链接：dxc -T ps_6_0 -entry main main.hlsl library.dxlib
```

**好处**：
- 跨链接边界优化

```
  □ □ ■ ■
  ↑ Helper lanes（填充 2×2 quad）

情况 2：discard 执行后
```

**检测**：
```hlsl
if (IsHelperLane()) {
```

## 三、开发者工作流增强

### 1. **Agility SDK + Independent Devices**

**问题**：
- 早期 D3D12 更新绑定 Windows 10 大版本更新
- 开发者难以确保用户有足够新的功能集

**Agility SDK 方案**：
```
应用自带 D3D12Core.dll
        ↓
D3D Loader 检查
        ↓
使用较新版本（应用自带 vs OS 内置）
```

**Independent Devices**：
- 解决插件无法指定 Agility SDK 的问题
- 通过 API 而非 exe export 指定

---

### 2. **Debug Layer Callback**

```cpp
// 类似 Vulkan 验证层回调
ID3D12InfoQueue1* infoQueue;
infoQueue->RegisterMessageCallback(
  CallbackFunction,
  D3D12_MESSAGE_CALLBACK_FLAG_NONE,
  nullptr,
  &callbackCookie
);
```

## 四、作者个人编程风格变化

### 1. **All-In On Bindless**

**"User-Space Bindings" 概念**：
```
传统：
  Descriptor Table → opaque handle
  需要 root signature 管理

Bindless (SM 6.6):
  uint32_t descriptorIndex → 用户完全控制
  可放入 struct、array、任何地方
```

**Root Signature 简化**：
```cpp
// 作者的 "Universal" Root Signature
D3D12_ROOT_PARAMETER1 params[] = {
  // 仅几个 CBV root descriptors
  CBV(register 0),  // Constant buffer
  CBV(register 1),  // ...
  CBV(AppSettings::CBufferRegister),
};
// 所有其他资源通过 ResourceDescriptorHeap 访问
```

**Shader 端**：
```hlsl
struct MeshConstants {
  float4x4 World;
  float4x4 View;
  float4x4 WorldViewProjection;
  float3 CameraPos;
  float Time;
  
  // Bindless descriptors!
  uint VertexBuffer;
  uint MeshletBuffer;
  uint MeshletBoundsBuffer;
  uint MeshletVerticesBuffer;
  uint MeshletTrianglesBuffer;
};

// 使用
StructuredBuffer<Vertex> vertices = 
  ResourceDescriptorHeap[constants.VertexBuffer];
```

## 五、关键洞察与建议

### 1. **API 设计哲学演变**

```
D3D11: "Driver knows best" → 驱动管理一切
D3D12 Early: "App is responsible" → 显式控制
D3D12 Now: "Explicit but convenient" → Enhanced Barriers, etc.
```

### 2. **Vulkan vs D3D12 融合趋势**

| 功能 | Vulkan | D3D12 |
|------|--------|-------|
| Barriers | 显式 sync/access/layout | Enhanced Barriers 趋同 |
| Bindless | Descriptor Indexing | ResourceDescriptorHeap |
| Ray Tracing | Ray Tracing extensions | DXR |
| Mesh Shaders | Mesh Shaders NV | Mesh Shaders |

### 3. **性能优化核心思路**

```
减少 CPU-GPU 同步
    ↓
GPU-Driven Rendering
    ↓
ExecuteIndirect / Work Graphs
    ↓
减少 Draw Call 开销
```
